Required owner: `growth-core` only.

---

## 1. No event is defined by this slice

`DecisionArtefact` is persisted state with **no consumer outside `growth-core` at MS-002**. Per the
delivery-plan slice scope rule, a `growth.decision.recorded.v1` event would be a speculative
integration with nothing subscribing to it, so none is specified here.

The schema in `schemas/` is therefore a **validation schema for the write API and the contract
tests**, not an event envelope. When S1b or S11 needs to react to decisions, an event wrapping this
same payload is added then — the payload shape is already pinned, so that addition is additive.

## 2. Artefact types

```ts
type ISODateTime = string;   // RFC 3339, UTC, e.g. "2026-07-19T14:03:11Z"
type Money = { value: string; currency: string };  // decimal STRING, ISO-4217 code

interface DecisionArtefactBase {
  artefactVersion:    1;
  decisionArtefactId: string;        // uuid, generated by the CLIENT (see §4)
  workspaceId:        string;
  experimentId:       string;
  experimentVersion:  string;
  evidenceReferences: string[];      // pointers only — NEVER inline personal data
  policyVersion:      string;
  decidedByType:      "human";       // always human at MS-002
  decidedById:        string;
  decidedAt:          ISODateTime;
  canonicalHash:      string;        // lowercase hex SHA-256, see §3
}
```

### 2.1 `experiment.launch`

```ts
interface ExperimentLaunchDecision extends DecisionArtefactBase {
  decisionType: "experiment.launch";
  hypothesis:   string;              // what we believe, in the owner's words
  rationale:    string;              // why now, why this budget, why this audience
  plannedAction: {
    platform:  "google_ads";
    budgetCap: Money;
    startAt:   ISODateTime;
    endAt:     ISODateTime;
  };
}
```

### 2.2 `experiment.stop`

```ts
interface ExperimentStopDecision extends DecisionArtefactBase {
  decisionType: "experiment.stop";
  reason:       string;              // REQUIRED, non-blank — see §5
  stoppedAt:    ISODateTime;         // when spend actually stops, not when the call was made
}
```

Carries no `hypothesis` and no `plannedAction`: nothing is being proposed. It binds to its launch
through `experimentId` + `experimentVersion`, so the pair reads as one story.

### 2.3 `experiment.budget_change`

```ts
interface ExperimentBudgetChangeDecision extends DecisionArtefactBase {
  decisionType:        "experiment.budget_change";
  reason:              string;       // REQUIRED, non-blank
  supersedesArtefactId: string;      // artefact that established the CURRENT cap
  previousBudgetCap:   Money;        // must equal the cap in supersedesArtefactId
  newBudgetCap:        Money;
  effectiveFrom:       ISODateTime;
}

type DecisionArtefact =
  | ExperimentLaunchDecision
  | ExperimentStopDecision
  | ExperimentBudgetChangeDecision;
```

**`supersedesArtefactId` makes the cap chain verifiable rather than asserted.** Without it,
`previousBudgetCap` is a number the client claims and nothing can check. With it, the current cap
is reconstructable by walking `launch → change → change`, and a client that has drifted out of sync
with the stored state is rejected instead of silently writing a false history. This is the same
reason `effectKey` exists in S1b: the record must be checkable, not merely present.

`previousBudgetCap` is retained despite being derivable — it makes each artefact readable on its own,
three months later, without replaying the chain.

## 3. Canonical hash

Per [D-004 §3](../07_decisions/D-004-decision-artefact-shape-and-hash.md), canonicalisation is
**RFC 8785 (JCS)**.

```
input  = the artefact object with the `canonicalHash` key REMOVED entirely
         (removed — not set to null, not set to "")
bytes  = JCS(input)                       // RFC 8785: lexicographic key order by UTF-16
                                          // code unit, no insignificant whitespace,
                                          // ES6 number serialisation, UTF-8 output
hash   = SHA-256(bytes)
output = lowercase hex, 64 chars, no prefix
```

Pinned details that two implementations would otherwise diverge on:

| Rule | Value |
|---|---|
| `canonicalHash` in the hash input | **absent** — key deleted before canonicalisation |
| Optional fields that are absent | **omitted**, never emitted as `null` |
| `evidenceReferences` ordering | **preserved as given** — it is a list, not a set; JCS does not reorder array elements |
| Money `value` | decimal string, already canonical; never converted to a number |
| Timestamps | RFC 3339, **UTC, `Z` suffix**, no offsets like `+02:00`, no fractional seconds |
| Empty `evidenceReferences` | `[]`, never omitted |

**Timestamps are normalised before hashing, not after.** `2026-07-19T16:03:11+02:00` and
`2026-07-19T14:03:11Z` are the same instant and must produce the same hash; JCS operates on strings
and would give them different ones. Normalisation to UTC `Z` is therefore part of *building* the
artefact, and a non-`Z` timestamp is rejected at the API boundary (§5) rather than quietly accepted.

Implementations verify against RFC 8785's own test vectors before verifying against each other.

## 4. Write API

```
POST /governance/decisions
Content-Type: application/json
```

Body: exactly one artefact. No batching — decisions are low-volume, deliberate acts.

`decisionArtefactId` is **client-generated**, which is what makes retries safe: the same decision
submitted twice carries the same id and lands once.

`canonicalHash` may be supplied or omitted. The server always computes the hash itself and stores
its own value. If the client supplied one that differs, the request is **rejected** — that mismatch
is the cross-implementation divergence check from §3 firing in production, and accepting the server
value silently would hide exactly the bug this contract exists to prevent.

### Responses

| Code | Meaning | Client behaviour |
|---|---|---|
| `201 Created` | Artefact committed | Done |
| `200 OK` | `decisionArtefactId` already present, **identical content** | Done, do not retry |
| `409 Conflict` | `decisionArtefactId` already present, **different content** | Do not retry; this is a bug or an edit attempt |
| `422 Unprocessable Entity` | Validation failure (§5), including hash mismatch | Do not retry unchanged |

**`409`, not overwrite, is how immutability is enforced at the API.** §6 enforces it again at the
database, because an application-layer guarantee disappears the moment a second writer exists.

### Read

```
GET /governance/decisions?experimentId=<id>
```

Returns the artefacts for an experiment ordered by `decidedAt` ascending, so launch → budget
changes → stop reads as one story. This is the query behind the owner manual check in F-001.

## 5. Validation rules

All violations return `422`.

| # | Rule |
|---|---|
| V1 | `decisionType` must be one of the three declared values |
| V2 | Fields not permitted for the given `decisionType` are rejected (`additionalProperties: false`) — a `stop` carrying `plannedAction` or `hypothesis` fails |
| V3 | `reason` on `stop` and `budget_change`: missing, empty, or whitespace-only is rejected, **not defaulted** |
| V4 | `hypothesis` and `rationale` on `launch`: same non-blank rule |
| V5 | A `stop` or `budget_change` whose `experimentId` + `experimentVersion` has no prior `experiment.launch` is rejected |
| V6 | `budget_change.supersedesArtefactId` must exist, belong to the same `experimentId` + `experimentVersion`, and be the artefact currently establishing the cap |
| V7 | `budget_change.previousBudgetCap` must equal the cap established by `supersedesArtefactId`, value and currency |
| V8 | All `Money.currency` values within one experiment must match the launch currency |
| V9 | Timestamps must be RFC 3339 UTC with a `Z` suffix (§3) |
| V10 | `plannedAction.endAt` must be strictly after `startAt` |
| V11 | Supplied `canonicalHash` must equal the server-computed hash (§4) |
| V12 | `decidedByType` must be `"human"` at MS-002 |

**Blank is rejected rather than defaulted throughout.** A defaulted `reason` produces a record that
looks complete and carries no information, which is worse than a missing one — it is not visibly
absent when the dataset is read six months later.

## 6. Storage

```sql
CREATE SCHEMA IF NOT EXISTS governance;

CREATE TABLE governance.decision_artefact (
  decision_artefact_id uuid        PRIMARY KEY,
  workspace_id         text        NOT NULL,
  experiment_id        text        NOT NULL,
  experiment_version   text        NOT NULL,
  decision_type        text        NOT NULL
                         CHECK (decision_type IN
                           ('experiment.launch','experiment.stop','experiment.budget_change')),
  artefact_version     int         NOT NULL,
  body                 jsonb       NOT NULL,   -- full artefact, canonical_hash included
  canonical_hash       char(64)    NOT NULL,
  decided_by_id        text        NOT NULL,
  decided_at           timestamptz NOT NULL,
  supersedes_id        uuid        REFERENCES governance.decision_artefact(decision_artefact_id),
  created_at           timestamptz NOT NULL DEFAULT now()
);

-- the read query behind GET /governance/decisions
CREATE INDEX decision_artefact_experiment_idx
  ON governance.decision_artefact (experiment_id, experiment_version, decided_at);

-- one launch per experiment version
CREATE UNIQUE INDEX decision_artefact_one_launch_idx
  ON governance.decision_artefact (experiment_id, experiment_version)
  WHERE decision_type = 'experiment.launch';

-- a given cap is superseded at most once: no forked budget history
CREATE UNIQUE INDEX decision_artefact_supersedes_idx
  ON governance.decision_artefact (supersedes_id)
  WHERE supersedes_id IS NOT NULL;
```

### Immutability enforced in the database

```sql
CREATE OR REPLACE FUNCTION governance.reject_artefact_mutation()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  RAISE EXCEPTION 'decision_artefact is append-only (attempted %)', TG_OP;
END;
$$;

CREATE TRIGGER decision_artefact_immutable
  BEFORE UPDATE OR DELETE ON governance.decision_artefact
  FOR EACH ROW EXECUTE FUNCTION governance.reject_artefact_mutation();
```

**A trigger, not an ORM convention.** Prisma can be bypassed by a migration, a psql session, or the
next service to reach this table; the trigger cannot. Immutability that depends on every future
caller behaving is not immutability. Corrections create a new artefact — the record of the mistake
is itself part of the audit trail.

The two partial unique indexes encode rules V5–V7 as constraints rather than as application checks,
so a race between two concurrent writers cannot produce two launches or a forked cap chain.

`decision_artefact_one_launch_idx` is scoped to `experiment_version`: a genuinely new launch means a
new version, which is how the model already expresses "we are trying something different."

## 7. PII and erasure

The artefact holds **references only**. `evidenceReferences` are pointers to pseudonymous
touchpoints; identity is reachable exclusively through the erasable `IdentityLink`
([architecture §7.9](../06_architecture/ARCHITECTURE.md), [C-005 §6](C-005-landing-and-ingestion.md)).

| Data | Retention | On erasure request |
|---|---|---|
| `governance.decision_artefact` | **indefinite** — operational, not personal | untouched |
| `IdentityLink` referenced by it | 14 months or on request | severed; artefact remains readable |

Immutability and the right to erasure are in direct conflict, and this is the seam that resolves
them: a deletion request severs the link, never the decision history. An artefact that inlined an
email would make the two requirements genuinely irreconcilable.

Consequence for `evidenceReferences`: a reference must resolve to something *pseudonymous*. A
pointer whose identifier is itself personal data (an email used as a key) reintroduces the problem
it was meant to avoid.

## 8. Executable contract tests

Per delivery-plan §8. Covers the F-001 validation plan plus what D-004 added.

| Test | Asserts |
|---|---|
| Schema conformance | Every artefact `growth-core` writes validates against `decision-artefact.v1.json` |
| Consumer tolerance | The reader accepts everything the schema permits, optional fields absent |
| Immutability (app) | `POST` with an existing id and different content → `409` |
| Immutability (db) | Direct `UPDATE` and `DELETE` both raise; row unchanged |
| Idempotency | Same artefact posted twice → one row, second returns `200` |
| Hash stability | Same artefact canonicalises to the same hash across runs, processes and a fresh node instance |
| Hash key-order independence | Two JSON bodies differing only in key order produce the same hash |
| Hash timezone independence | `+02:00` input is rejected (V9); the UTC equivalent hashes stably |
| Hash sensitivity | Any single field change produces a different hash |
| Hash self-exclusion | Hash computed with `canonicalHash` present vs absent — contract requires absent; a present-key implementation is detectably wrong |
| JCS vectors | The canonicaliser passes RFC 8785's published test vectors |
| Stop requires reason | Missing, empty or whitespace-only `reason` → `422` |
| Stop shape | `stop` carrying `plannedAction` or `hypothesis` → `422` |
| Stop references a launch | `stop` with no prior `launch` for that `experimentId` + `experimentVersion` → `422` |
| Budget chain | `previousBudgetCap` ≠ the cap in `supersedesArtefactId` → `422` |
| Budget fork | Two changes superseding the same artefact → second rejected by the unique index |
| Currency consistency | A change in a different currency from the launch → `422` |
| Concurrent launch | Two simultaneous launches for one experiment version → exactly one succeeds |
| No PII | Stored artefacts contain no email, phone or name — scanned by pattern |
| Erasure survival | Deleting the referenced `IdentityLink` leaves the artefact intact and readable |

## 9. Vault keys added by this slice

None. S1a introduces no secret — the hash is unkeyed by design
([F-001](../10_features/F-001-decision-record-and-governance.md): a hash detects mutation, a
signature would defend a door that is already open).
