## 1. Event envelope

Follows the existing ecosystem convention. Type naming from `orders-microservice` (`<service>.<aggregate>.<action>.v<n>`), envelope fields from `leads-microservice.LeadLifecycleEvent`.

```ts
interface GrowthEventEnvelope<TPayload extends object> {
  eventId:        string;   // uuid — idempotency key, generated by the producer
  eventType:      string;   // e.g. "growth.touchpoint.observed.v1"
  eventVersion:   number;   // 1
  occurredAt:     string;   // ISO 8601, when the fact happened (not when received)
  producer:       string;   // "growth-web" | "bazos-service" | "growth-core"
  workspaceId:    string;   // aggregate-root scope — never `tenantId`
  correlationId:  string;   // ties one user journey together
  causationId?:   string;   // OPTIONAL — root events have none
  dataClass:      string;   // "anonymous" | "personal" | "operational"
  payload:        TPayload;
}
```

> ⚠️ **Ecosystem inconsistency noted, not fixed here.** `orders-microservice` names this field `source`; `leads-microservice` names it `producer`. Growth follows `producer`, matching architecture §4.6 and `leads` — which is in this slice. Reconciling `orders` is out of scope and should not be done incidentally.

`dataClass` drives the retention job: `anonymous` → 14 months, `personal` → severed on erasure request, `operational` → indefinite.

## 2. Events

### 2.1 `growth.touchpoint.observed.v1`

Producer `growth-web` · consumer `growth-core` · `dataClass: "anonymous"`

```ts
interface TouchpointObservedPayload {
  sessionId:          string;         // uuid, issued only after consent
  experimentId:       string;
  experimentVersion:  string;
  landingVersionId:   string;         // immutable; a change creates a new id
  gclid?:             string;
  fbclid?:            string;         // reserved, unused at stage 1
  utm?: {
    source?: string; medium?: string; campaign?: string;
    term?: string;   content?: string;
  };
  referrer?:          string;
  consentEvidence: {                  // REFERENCE, never a copied consent record
    consentRecordId:   string;
    consentVersion:    number;
    applicablePurposes: string[];
    statusAtEventTime: "granted" | "denied" | "withdrawn" | "not_required";
    evaluatedAt:       string;
  };
}
```

**No contact details at this stage.** The visitor is anonymous; `sessionId` is the only handle.

### 2.2 `growth.registration.completed.v1`

Producer `bazos-service` · consumers `growth-core`, `leads-microservice` · `dataClass: "personal"`

```ts
interface RegistrationCompletedPayload {
  registrationId: string;
  gsid?:          string;   // signed session token, raw as received; absent when consent was refused
  gsidSource?:    "cookie" | "query";
  email:          string;
  phone?:         string;   // E.164
  registeredAt:   string;
}
```

`gsid` is passed **raw and unverified** — `bazos-service` does not hold the signing secret. Verification is `growth-core`'s job (§4).

**`gsid` absent is a valid, expected case**, not an error: consent refused, cookie cleared, or a direct visit. The registration is still emitted.

### 2.3 `growth.lead.created_from_registration.v1`

Producer `leads-microservice` · consumer `growth-core` · `dataClass: "personal"`

```ts
interface LeadCreatedFromRegistrationPayload {
  leadId:         string;
  registrationId: string;
  sourceService:  "bazos-service";
  createdAt:      string;
}
```

Anchors `LeadQualificationEvent` (D19) — qualification attaches to this lead as a post-hoc quality assessment, not as the experiment's primary metric.

### 2.4 `growth.spend.observed_manual.v1`

Producer `growth-core` (owner-entered) · `dataClass: "operational"`

```ts
interface ManualSpendObservationPayload {
  observationId:     string;
  experimentId:      string;
  platform:          "google_ads";
  periodStart:       string;         // ISO date
  periodEnd:         string;
  amount: { value: string; currency: string };   // decimal STRING, never float
  evidenceReference: string;         // provider report / export identifier
  enteredBy:         string;
  enteredAt:         string;
  isManual:          true;           // literal — never presented as invoice-reconciled
}
```

Money as a decimal **string**. IEEE-754 floats silently lose cents; a currency amount must never round-trip through a double.

## 3. Ingestion endpoint

```
POST /ingest/events
Content-Type: application/json
```

Body: a single envelope, or an array (max 50).

### Responses

| Code | Meaning | Client behaviour |
|---|---|---|
| `202 Accepted` | **Committed to the buffer.** Only returned after the transaction commits | Done |
| `200 OK` | `eventId` already present — duplicate | Done, do not retry |
| `400 Bad Request` | Schema violation | Do **not** retry; log |
| `413 Payload Too Large` | Batch > 50 or body > 256 KB | Split |
| `503 Service Unavailable` | Buffer unwritable | Retry with backoff, **same `eventId`** |

**`202` is never returned before the commit.** Acknowledging first and writing after is precisely the loss this slice exists to prevent.

Browser retries reuse the same `eventId` — that is what makes `200` on duplicate safe.

## 4. `gsid` signing scheme

```
gsid       = base64url(sessionId) + "." + base64url(HMAC-SHA256(sessionId, secret))
secret     = Vault  secret/prod/growth  →  GROWTH_GSID_HMAC_SECRET   (32 bytes, random)
cookie     = Domain=bazos.alfares.cz · Path=/ · Secure · SameSite=Lax · Max-Age=7776000  (90d)
```

**Verification** in `growth-core`, constant-time comparison:

| Outcome | Action |
|---|---|
| Signature valid, `sessionId` known | Create `IdentityLink` → attribution complete |
| Signature valid, `sessionId` unknown | Registration recorded, attribution absent; increment `gsid_orphan` counter |
| Signature invalid | Registration recorded, attribution **dropped**; increment `gsid_forged` counter |
| `gsid` absent | Registration recorded, attribution absent — **expected path, not an error** |

A non-zero `gsid_forged` counter means someone is editing attribution parameters. It must be visible in monitoring, because the corrupted signal would otherwise flow straight into budget decisions.

The secret is generated once and stored in Vault. It is never in source, never logged, never sent to the browser — only the derived signature is.

## 5. Buffer

```sql
CREATE SCHEMA IF NOT EXISTS ingest;

CREATE TABLE ingest.event_buffer (
  event_id      uuid        PRIMARY KEY,
  workspace_id  text        NOT NULL,
  event_type    text        NOT NULL,
  event_version int         NOT NULL,
  payload       jsonb       NOT NULL,           -- full envelope
  received_at   timestamptz NOT NULL DEFAULT now(),
  status        text        NOT NULL DEFAULT 'pending'
                            CHECK (status IN ('pending','published','failed','dead')),
  attempts      int         NOT NULL DEFAULT 0,
  last_error    text,
  published_at  timestamptz
);

-- drain query touches only unprocessed rows
CREATE INDEX event_buffer_pending_idx
  ON ingest.event_buffer (received_at)
  WHERE status IN ('pending','failed');

-- retention sweep
CREATE INDEX event_buffer_published_idx
  ON ingest.event_buffer (published_at)
  WHERE status = 'published';
```

`event_id` as primary key gives idempotency from the database constraint, not from application logic — a duplicate insert fails on conflict rather than needing a read-then-write race.

### Worker

```
SELECT … WHERE status IN ('pending','failed') AND attempts < 10
ORDER BY received_at
FOR UPDATE SKIP LOCKED
LIMIT 100
```

`SKIP LOCKED` allows more than one worker without double publishing.

| Result | Transition |
|---|---|
| Published to RabbitMQ | `published`, `published_at = now()` |
| Transient failure | `failed`, `attempts += 1`, backoff `min(2^attempts, 300)` s |
| `attempts` reaches 10 | `dead`, alert raised |

**`dead` rows are never deleted automatically.** A row that cannot be published stays visible for inspection — silent loss here defeats the buffer's purpose.

## 6. Retention

| Data | Period | Action |
|---|---|---|
| `ingest.event_buffer` where `status='published'` | 30 days | delete |
| `ingest.event_buffer` where `status='dead'` | — | **never auto-delete** |
| `AnonymousTouchpoint` | **14 months** | hard delete |
| `IdentityLink` | 14 months, or immediately on erasure request | hard delete |
| Aggregated rollups (registrations/day, cost per registration) | indefinite | keep — not personal data |

Daily job at 03:00 Europe/Prague. Every run logs counts per category; a run that deletes nothing logs that too, so a silently broken job is visible.

**On the period:** 14 months is chosen by *purpose* — the longest genuine need is a year-over-year comparison of a seasonal experiment. GDPR sets no maximum retention; Article 5(1)(e) requires data be kept no longer than necessary, so "as long as permitted" is not a target. Anything beyond 14 months needs a documented purpose.

## 7. Executable contract tests

Per delivery-plan §8 — schemas in a shared package, verified on both sides. A contract document never fails; an executable schema does.

| Test | Side | Asserts |
|---|---|---|
| Producer conformance | `growth-web`, `bazos-service`, `leads-microservice` | every emitted event validates against its schema |
| Consumer tolerance | `growth-core` | parser accepts everything the schema permits, including optional fields absent |
| Idempotency | `growth-core` | same `eventId` twice → one row, second returns `200` |
| Ack ordering | ingestion endpoint | `202` never precedes commit (failure injection between write and response) |
| Failure injection | end-to-end | `growth-core` down during registration → event lands after recovery |
| Consent refusal | end-to-end | no consent → no `gsid` → registration succeeds, attribution absent, **no error** |
| Signature forgery | `growth-core` | edited `gsid` → attribution dropped, `gsid_forged` incremented |
| Retention | job | touchpoints > 14 months deleted; aggregates and `dead` rows survive |
